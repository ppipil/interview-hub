## Context

The repository already has product requirements in `docs/PRD.md`, an interface contract in `docs/openapi.yaml`, and read-only visual references in `stitch_exports/`. What is missing is the frontend foundation layer that future pages can depend on: shared domain types, a session-aware mock API, and a global store that owns flow state instead of spreading it across page components.

## Goals / Non-Goals

**Goals:**

- Keep frontend domain models aligned with `docs/openapi.yaml`
- Provide mock APIs that behave like backend endpoints, including latency and session progression
- Centralize setup, interview, and feedback workflow state in Zustand
- Make later page work depend on services and store instead of inline fake data

**Non-Goals:**

- Building final page components
- Adding a real backend transport layer
- Introducing authentication, persistence, or analytics

## Decisions

### Use `docs/openapi.yaml` as the source contract for TypeScript models

This keeps request and response shapes stable before the backend exists. The alternative was to let the store define ad hoc frontend-only types, but that would increase drift and rework once the real API lands.

### Implement the mock backend as an in-memory session simulator

`src/services/api.ts` will own static interviewer fixtures, generated questions, in-memory session maps, and feedback generation. The alternative was to hardcode fake payloads in each page or store action, which would make behavior inconsistent and difficult to replace with real HTTP later.

### Put async workflow orchestration in Zustand actions

The store will call the mock service layer and keep loading, success, and error state close to the workflow state. The alternative was to keep the store synchronous and let pages coordinate service calls themselves, but that would leak orchestration logic into the UI layer before the feature components even exist.

## Risks / Trade-offs

- [Mock behavior diverges from the future backend] -> Keep the domain types and service signatures locked to `docs/openapi.yaml`, and treat the mock layer as replaceable infrastructure
- [Store becomes too page-specific too early] -> Limit the store to workflow state and async actions, and keep rendering concerns out of it
- [OpenSpec artifacts become stale] -> Update the same capability when future frontend requirements affect typed contracts, mock behavior, or flow orchestration

## Migration Plan

1. Add the OpenSpec artifacts for the frontend foundation change
2. Introduce shared types under `src/types/index.ts`
3. Introduce the mock service layer under `src/services/api.ts`
4. Introduce the Zustand workflow store under `src/store/useInterviewStore.ts`
5. Let Phase 4 pages depend on this foundation instead of writing local mock state

Rollback is straightforward because this change only adds new frontend foundation files and does not migrate persisted data.

## Open Questions

- Whether the final app routes will derive navigation entirely from URL state or partially from store stage state
- Whether guided mode hints should be generated by the mock API or remain presentation-only until the real backend is connected
