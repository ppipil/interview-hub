# AI 模拟面试 MVP PRD

## 1. 项目定位

这是一个从 0 到 1 的 AI 模拟面试 MVP。产品要在尽量短的时间里，让用户完成一次有代入感的“闯关式”面试练习：选配置、开面、实时作答、拿反馈，再决定要不要马上再来一轮。

MVP 只验证一件事：

- 用户是否愿意用 AI 面试官或 AI 分身面试官完成一次完整练习

## 2. 产品目标

- 让用户在 3 分钟内走完一轮完整流程
- 让用户清楚理解“岗位 + 模式 + 面试官”三个配置维度
- 让结果页产出足够具体的反馈，驱动用户继续挑战下一轮

已确认的下一阶段工作重点如下：

- 题库存储 MVP
  - 先落数据库表结构
  - 先支持通用题库入库与后续复用

补充说明：

- AI 分身面试官的目标接入方式是 `SecondMe Visitor Chat` 的开发者应用模式

- 当前代码已补 Visitor Chat 路径；测试期通过 `.env` 的 `SECONDME_AVATAR_API_KEY` 或数据库 `interviewer_profiles` 提供目标分身 key
- 应用内面试官 skill 由 Interview Hub 数据库维护，并在提问 prompt 中优先拼入；SecondMe 平台侧 skill 管理能力不作为当前阻塞项
- 当前已补隐藏管理台，可通过页面输入或上传 `.txt` / `.md` skill 文件，避免直接手写 SQL
- OAuth 登录检查仅用于验证 SecondMe 授权返回字段，不作为 MVP 面试主流程依赖

## 3. 范围定义

### 本版本必须实现

- 首页入口
- 岗位选择，固定 5 个岗位
- 模式选择，固定 2 种模式
- 面试官选择，支持系统面试官和 AI 分身面试官
- 面试流程，支持可配置轮次，目标范围为 1 到 10 轮，默认 3 轮
- 结果反馈，至少包含总结、3 条建议、参考优化方向
- 再来一轮，形成闭环

当前状态：

- 首页入口：`已完成`
- 固定 5 个岗位：`已完成`
- 固定 2 种模式：`已完成`
- 面试官选择（系统面试官 + AI 分身面试官）：`部分完成`
  当前前后端已补系统面试官与 AI 分身面试官的 provider 骨架，并提供隐藏管理台维护面试官资料；豆包与 SecondMe Visitor Chat 仍待用真实凭证完成联调
- 面试流程（支持可配置轮次）：`已完成`
  当前前后端已支持显式 `1-10` 轮选择与统一校验，默认仍为 `3` 轮
- 结果反馈（总结、3 条建议、参考优化方向）：`部分完成`
  总结与建议已完成，反馈页已补每轮评价与参考答案；整体视觉与信息密度仍可继续打磨
- 再来一轮闭环：`部分完成`
  流程已可重新开始，但反馈页仍缺少更明确的主 CTA 承接

### 本版本不做

- 用户系统与登录鉴权
- 历史记录和成长曲线
- 语音输入输出
- 复杂评分算法
- 课程系统和 marketplace
- 多轮复杂面试编排

## 4. 核心用户流

1. 用户进入首页，被“开始挑战”CTA 引导进入设置页。
2. 用户在设置页选择岗位、模式和面试官。
3. 用户点击“开始挑战”，前端初始化面试 Session。
4. 用户进入真实模拟对话页，完成本轮选择的 1 到 10 轮问答。
5. 系统在面试结束后生成反馈，用户跳转到结果反馈页查看总结与建议。
6. 用户点击“再来一轮”，重新回到设置页继续练习。

## 5. 页面定义

作为产品入口，快速建立“AI 模拟面试”的价值感和挑战感。

核心功能：

- 展示产品标题、副标题和核心卖点
- 提供清晰的主 CTA：“开始挑战”
- 点击主 CTA 后跳转 `/setup`

当前状态：

- `已调整范围`
- 当前首页已按最新设计收缩为“一句文案 + 一个开始挑战按钮”
- 主 CTA 已完成并可进入设置页
- 副标题与卖点区未保留，属于有意简化而非遗漏

### 5.2 面试设置页 `Interview Setup`

页面目标：
让用户在最短路径内完成本轮挑战配置，并对本轮难度和风格形成预期。

核心功能：

- 岗位选择
  - 固定岗位：前端工程师、后端工程师、产品经理、运营、数据分析
- 模式选择
  - 引导模式 `guided`
  - 真实模式 `real`
- 面试官选择
  - 系统面试官：如技术面试官、HR 面试官
  - AI 分身面试官：先固定 2 个样例
- 轮次选择
  - 默认 `3` 轮
  - 允许用户选择 `1-10` 轮
- 配置摘要
  - 在 CTA 旁或 CTA 上方展示当前选择，减少误操作
- 开始挑战
  - 点击后调用初始化 Session 接口
  - 成功后携带 `sessionId` 和首个问题进入 `/interview`
- 异常提示
  - 配置不完整或接口失败时，展示明确提示并支持重试

建议状态结构：

```json
{
  "role": "frontend",
  "mode": "guided",
  "interviewerId": "system_tech"
}
```

### 5.3 真实模拟对话页 `Live Interview`

页面目标：
营造连续、专注、低打断的答题体验，让用户感觉自己真的在和面试官过招。

核心功能：

- 展示面试官信息、岗位、模式、当前轮次
- 以消息流形式展示问题和回答历史
- 提供回答输入框与发送按钮
- 发送回答后进入 loading 状态，避免重复提交
- 根据接口返回追加 AI 下一问
- 会话结束后进入“可查看反馈”的收口状态
- 异常情况下保留用户输入并允许重试

建议状态结构：

```json
{
  "sessionId": "uuid",
  "currentRound": 1,
  "totalRounds": 3,
  "messages": []
}
```

### 5.4 结果反馈页 `Feedback Report`

页面目标：
把一次练习转化为可读、可执行、能驱动下一轮挑战的结果卡片。

核心功能：

- 总体评价
  - 1 段简洁总结，概括本轮表现
- 维度反馈
  - 至少 3 个维度，如表达清晰度、专业深度、问题贴合度
- 改进建议
  - 至少 3 条具体建议，避免空泛表述
- 参考优化方向
  - 给出更优回答思路或简短示例
- 再来一轮
  - 点击后返回设置页

建议数据结构：

```json
{
  "summary": "整体表达在线，但细节深度还可以继续补强。",
  "improvements": [
    "补足技术方案的取舍理由",
    "回答里增加量化结果",
    "结尾补一句总结会更稳"
  ],
  "suggestedAnswer": "如果重新回答，我会先讲背景，再讲方案、取舍和结果。"
}
```

## 6. MVP 接口约束

- 前端以 `sessionId` 作为流程主键
- 初始化接口需要直接返回首个问题，减少额外请求
- 对话接口同时负责提交回答和接收 AI 回复
- 反馈接口可独立拉取，便于刷新页面后恢复结果
- MVP 默认无用户登录态，后续预留 `Authorization` 扩展位
- AI 分身面试官的目标接入方式是通过后端完成 `SecondMe Visitor Chat` 开发者应用认证，再以匿名访客身份访问目标分身
- 测试期目标分身由项目方手动生成 avatar API key 并配置到后端；当前可先通过隐藏管理台写入 `interviewer_profiles`，后续补正式后台权限后支持多分身维护
- 系统面试官与 AI 分身面试官允许使用不同的上游 provider 与不同的 Prompt 编排策略
